# AGENT.md

## 文档定位

本文件是本项目的长期工作基线，用于保证产品、原型、设计、开发、测试和交付在上下文压缩、长时间中断或人员变更后仍保持一致。

任何参与本仓库工作的 AI Agent 或贡献者都必须遵守本文件。若具体需求与本文件冲突，应先明确指出冲突及影响，再采用用户最新的明确决定；不得静默忽略规则。

## 强制重读规则

每次发生对话上下文压缩、摘要、重置、部分丢失、模型切换或任务恢复时，Agent 必须在继续任何工作之前，从头到尾重新阅读根目录的 `AGENT.md`。

这是硬性要求，不得依赖压缩前的记忆，也不得只阅读摘要或部分章节。未重新完整阅读前，不得继续分析、设计、编码、修改文件或给出实施结论。

开始较大范围工作前，也应先确认当前方案与本文件一致。

## 沟通语言

- 本项目后续所有对话、进度说明、问题确认、交付总结和代码解释默认使用简体中文。
- 原型中的业务文案、需求说明和文档默认使用简体中文。
- 代码标识符、框架约定、命令、文件名、接口字段及业内通用技术名词可保留英文。
- 用户明确要求使用其他语言时，以该次要求为准；后续未再次说明时恢复使用简体中文。
- 表达应具体、直接、可验证。不得使用“按常规处理”“大概如此”等模糊表述替代产品规则。

## 项目定位

本项目是一套可在线部署的需求与交互原型展示系统，服务于产品、开发、测试及相关干系人的协作。

它不仅展示页面外观，还应以 Web 原生方式提供接近或优于 Axure 的说明精度，将以下内容集中在同一处：

- 需求管理入口
- 可交互页面原型
- 页面级和组件级结构化说明
- 原型组件与说明的精确关联
- 可供开发和测试访问的线上部署版本

最终目标是让开发和测试无需猜测即可理解页面、字段、状态、边界和交互逻辑。

## 交付形式与数据策略

- 默认交付形式为前端 Web 应用，不以静态图片、纯文档或 Axure 导出文件作为主要交付物。
- V1 优先构建可静态部署的网站，使用 JSON、TypeScript 或 JavaScript 配置维护需求、页面、组件和说明数据。
- V1 默认不引入数据库，展示内容不得无必要地依赖后端服务。
- 视图渲染与需求数据应分离，产品内容不应散落或深埋在 UI 组件中。
- 数据结构应集中、清晰，并尽量通过类型或 Schema 校验，方便产品人员维护。

仅在明确出现以下需求时再评估数据库和服务端：

- 多人在线编辑及并发协作
- 登录、权限或数据隔离
- 评论、审批、发布流程
- 在线编辑而非代码或配置文件维护
- 产品内历史版本查询、差异比较或审计
- 需要保存真实用户操作数据

## 信息架构

V1 至少包含：

- 需求列表页
- 需求详情页
- 需求内页面导航（位于高保真原型区内部）
- 左侧高保真原型展示区
- 右侧结构化说明面板

需求详情页默认采用稳定的两栏结构：

- 左侧：高保真原型展示区，约占页面总宽度的 80%，是页面的视觉和操作主体。
- 右侧：结构化说明面板，约占页面总宽度的 20%。
- 返回需求列表的按钮必须位于右侧说明面板最顶部，位置稳定且清晰可见。
- 需求包含多个原型页面时，页面导航置于左侧高保真区顶部或内部，不额外占用独立栏位。

不同需求应尽量保持相同的信息结构，降低开发和测试的理解成本。变更日志、评论、版本对比、导出和评审状态属于后续可选模块。

## 需求列表规范

进入原型详情前必须提供需求列表。每条需求至少包含：

- 需求编号
- 需求主题
- 产品负责人
- 开始时间
- 完成时间

默认产品负责人为 `Mike`。建议同时包含：需求状态、优先级、最后更新时间、简要说明和查看操作。

规则如下：

- 点击需求行或明确的查看入口进入对应需求详情。
- 需求编号必须唯一、稳定，不因排序或标题变化而改变。
- 未开始或未完成时，对应日期允许为空，但必须展示统一占位符，不得伪造日期。
- 需求状态至少应区分待开始、进行中、已完成；若增加其他状态，必须定义流转规则。
- 日期展示格式必须在全站保持一致。

## 需求详情规范

每个需求详情至少应展示：

- 需求编号、主题、负责人、状态和时间信息
- 需求目标、范围及必要的非目标说明
- 页面导航
- 交互原型
- 页面级说明
- 组件级说明
- 必要的状态、异常和边界示例

若需求包含多角色或权限差异，必须明确当前展示角色，并可切换或清楚列出差异。

## 组件与说明关联规范

右侧说明必须与原型组件建立精确、可见、双向的映射。不得使用“左上角字段”“下面的按钮”等易失效的空间描述作为唯一定位方式。

每个有业务意义的组件必须拥有稳定 ID，并遵守：

- 原型组件附近显示 ID 标记，说明面板显示相同 ID。
- 点击原型组件时，右侧自动定位、展开并高亮对应说明。
- 点击右侧说明时，中间原型自动定位并高亮对应组件。
- 可行时支持双向悬停关联，但悬停不能替代点击和键盘操作。
- 选中、高亮和聚焦状态必须视觉清晰，且不能改变布局尺寸。
- 页面切换后应清除无效选中状态，或恢复该页面最后一次有效选择。
- 同一组件的 ID 一经用于评审或开发，不应随意修改；必须修改时需记录映射或变更说明。

组件引用的完整标识采用 `需求编号/页面编号/组件ID`，例如 `REQ-001/P01/F03`，确保跨需求、跨页面引用无歧义。界面内可简写为 `F03`。

## 组件 ID 规范

组件 ID 在页面内唯一，使用简短、稳定、可读的前缀加两位数字：

- `Fxx`：输入、选择或展示字段
- `Bxx`：按钮或明确命令
- `Txx`：表格
- `Cxx`：卡片或重复内容项
- `Mxx`：弹窗、抽屉
- `Pxx`：面板或区域
- `Sxx`：状态、统计或摘要块
- `Nxx`：导航项

例如：`F01` 需求编号、`F03` 开始时间、`B01` 查询按钮、`T01` 需求列表。

同一页面不得重复 ID。跨页面可重复简写 ID，但完整标识必须不同。组件被删除后，原 ID 原则上不立即分配给含义不同的新组件，避免历史说明混淆。

## 右侧说明面板规范

说明面板包含页面级说明和组件级说明，并明确展示当前选中的页面及组件。

页面级说明至少包含：

- 页面目的与适用角色
- 入口、前置条件和退出路径
- 主流程及关键分支
- 页面状态及状态转换
- 权限差异
- 上下游依赖
- 页面级异常、空状态和限制

重要组件的说明至少包含：

- 完整组件标识、名称和类型
- 用途及适用场景
- 是否必填、是否只读、是否可见
- 默认值、占位内容和数据来源
- 数据类型、长度、精度、单位和格式
- 可选项、排序及联动规则
- 输入、显示和提交规则
- 校验时机、校验条件及错误文案
- 点击、悬停、聚焦、键盘及禁用逻辑
- 最小值、最大值、空值、重复值和极端值等边界
- 加载、成功、失败、无权限和空数据表现
- 角色、权限和可见性条件
- 关联接口、字段或后端依赖（已知时）
- 验收条件及必要备注

没有对应内容的项目应明确标注“不适用”或“待确认”，不能用缺失内容暗示无规则。`待确认` 项必须集中可见，不能被当作已定稿内容交付。

## 日期与时间控件规范

任何日期、时间或时间范围控件都不得省略精度说明。每个控件必须明确：

- 仅日期还是日期时间
- 是否精确到时、分、秒
- 展示格式和提交或存储格式
- 时区及跨时区规则
- 开始与结束是否可以相等
- 可选最小值、最大值及最大跨度
- 默认值及首次进入行为
- 是否允许手动输入
- 快捷选项，如今天、近 7 天、近 30 天
- 清空、重置和取消行为
- 非法输入、缺失值和越界时的提示
- 边界包含关系，例如结束时间是否包含该秒或该日

推荐明确写为 `YYYY-MM-DD`、`YYYY-MM-DD HH:mm` 或 `YYYY-MM-DD HH:mm:ss`。未特别说明时不得由开发自行推断精度。

### 时间范围筛选默认规则（强制）

除非具体需求明确说明例外，所有用于数据筛选的时间范围组件统一遵守：

- 在同一个范围组件内同时选择开始时间和结束时间。
- 精确到秒，展示格式为 `YYYY-MM-DD HH:mm:ss`。
- 开始时间默认使用所选开始日期的 `00:00:00`。
- 结束时间默认使用当前日期和当前时刻；若用户选择历史结束日期，则保留用户明确选择的时分秒。
- 必须提供快捷选项：今日、昨日、本周、30 天、90 天、180 天。
- 开始和结束边界均包含；开始时间不得晚于结束时间。
- 清空、重置、手动输入、非法格式、越界及最大范围仍需在具体组件说明中明确。

## 输入框默认规则（强制）

除非具体字段明确要求保留首尾空格，所有文本输入框统一遵守：

- 用户输入在失焦、查询或提交前执行首尾空格去除（trim）。
- 不擅自删除输入内容中间的空格；中间空格是否允许由字段自身格式规则决定。
- 去除首尾空格后再执行必填、长度和格式校验。
- 原型右侧组件说明必须明确写出“提交前去除首尾空格”，不得只在实现代码中隐式处理。

## 数据列表分页规范（强制）

除非具体需求明确说明不分页，所有数据列表必须提供完整分页器，并遵守：

- 展示数据总条数。
- 提供每页数量下拉选项：`10 / 20 / 50 / 100 / 200`。
- 默认每页 `50` 条。
- 默认每页 `200` 条的数据表必须在表格上方和下方各展示一套分页器，方便长列表在顶部直接翻页；其他默认分页数量的数据表不增加顶部分页器。
- 提供上一页、连续页码、明确的当前页、下一页。
- 提供页码跳转输入框或跳转器。
- 切换每页数量后回到第 1 页并重新查询。
- 筛选条件变化并重新查询后回到第 1 页。
- 总条数为 0 时显示空状态；是否保留分页器由具体页面说明，但不得显示无意义的空表格行。

## 说明信息密度规范（公共）

右侧说明以业务决策和开发差异为中心，不重复开发人员从界面结构即可直接看出的基础信息。

- 未经具体需求特别指定，不默认说明必填、非必填、AND 关系等常规实现信息。
- 筛选、重置、分页翻页等常规控件不默认建立独立组件说明；只有存在业务例外、特殊默认值、权限、边界或非标准行为时才说明。
- 数据表格组件说明不重复列出表格可见字段；表格本身就是字段定义。说明只记录排序、状态、异常、权限、并发和特殊处理规则。
- 说明卡片主标题下不显示无业务价值的类型副标题；组件类型可保留在数据结构中供内部使用，但不作为默认可见文案。
- 页面目标和主流程只保留帮助开发理解业务的内容，不重复表格列、按钮名称或从页面布局即可理解的下一步。
- 所有“去除首尾空格”说明必须列出具体输入框标题，例如“订单号、会员账号、真实姓名、所属站点提交前去除首尾空格”，不得只写“提交时处理”。
- 说明中的基础常识和重复内容应删除，不能以“写得更详细”为理由增加阅读成本。

## 公共展示规则

- 金额按实际精度展示：有小数时保留业务要求的小数位，无小数时显示整数，不强制追加 `.00`。
- 金额字段相对同一行普通字段略大并加粗，确保金额是数据列表的主要扫描信息。
- 会员标签、挂起原因、系统审核结果、出款状态等具有分类语义的值，统一使用标签样式展示；不同标签可通过低饱和语义色区分，但不得引入新的主题色。
- 关联标签、会员等级等纵向复合信息应分行展示；代理用户名、代理编号、挂起时间、挂起人、挂起原因等不同含义的字段必须带标题并分行展示。
- 可交互的日期范围组件必须是真实可打开的范围选择器，快捷选项放在日期组件标题行右侧，避免重复占用筛选区空间。
- 联想输入字段获取焦点且无输入时，应展示该字段允许的全部下拉选项；输入后按输入内容过滤。
- 联想输入的下拉选项只能在输入框获得焦点后显示；未聚焦时必须隐藏，不得作为常驻内容展示在页面上。
- 数据表复合字段采用多行展示时，字段标题使用次级灰色，真实字段值使用主要黑色，避免标题和值视觉权重相同。
- 每个包含金额的数据表说明应包含简短金额规则：无小数时不显示小数，有小数时按实际精度显示。
- 时间范围筛选点击开始时间或结束时间区域时，都必须打开同一个双日历范围面板，使用户在一次弹层内同时选择开始和结束；不得先弹出第二组相似时间输入框作为中间步骤。
- 双日历范围面板仅在开始时间或结束时间区域获得点击焦点后显示；未触发时必须隐藏，点击确认或取消后必须立即关闭。时间范围控件宽度按两组日期时间值合理设置，不无意义占满三组以上筛选列。
- 所有数据表统一使用“上级代理”作为代理复合字段表头，不使用“代理信息”等近义名称。
- “上级代理”复合字段内部统一使用“代理用户名”和“代理编号”作为子字段标题，不重复添加“上级”前缀。
- 筛选区按钮顺序默认以“筛选”为第一项，主筛选按钮的可点击面积应明显大于重置、导出等次级按钮。
- 所有筛选组件说明必须明确：筛选按钮的可点击面积大于其他按钮一倍，可参考原型。
- 所有可滚动区域的滚动条宽度或高度保持 `15px`，避免滚动条过细影响操作。存在溢出时，滚动条轨道和滑块必须常态可辨识，不得只保留不可见但可拖动的交互区域。
- 所有数据表必须统一使用一条完整说明：“表格仅在内容超出对应可视区域时显示滚动条：横向滚动条高度、纵向滚动条宽度均保持 `15px`；内容可完整显示时对应滚动条隐藏”。不得拆成横向、纵向两句，不得只说明其中一个方向，也不得把该规则放入筛选组件说明代替。
- 会员 VIP 等级筛选如无特殊业务规则，说明统一写为“读取VIP等级列表即可”，不在原型中固化具体等级数量。
- 左侧一级菜单和二级菜单的字号应保持同级可读性，不得因层级而把一级菜单做得更小。
- 页面局部说明性绿色眉标题如果不承载独立信息，不显示在页面主标题上方。

原型应达到开发能够理解布局、数据、状态和意图而无需猜测的程度。无需等同生产代码，但必须包含：

- 真实或具有代表性的页面结构和示例数据
- 关键表单、操作入口和反馈
- 理解业务所必需的可操作流程
- 加载、空数据、禁用、错误、成功等关键状态
- 弹窗、抽屉、确认、取消及返回行为
- 防重复提交、未保存离开等与场景相关的逻辑

不得只画默认态，也不得用无意义占位框代替影响实现判断的内容。原型中的按钮和控件若可见，应按说明可操作；尚未实现的操作必须明确标注。

## 视觉、响应式与可访问性

- 优先保证信息清晰、结构稳定和可扫描性，不以装饰牺牲理解效率。
- 桌面端是开发评审的主要场景，同时不得在常见窄屏下出现文字遮挡、面板不可访问或核心操作丢失。
- 桌面端需求详情默认保持约 `80% : 20%` 的原型区与说明区比例；说明区应具备可用的最小宽度，但不得无故挤占原型主区。
- 窄屏下两栏可改为上下布局、抽屉或标签切换，但返回入口和组件与说明的关联必须保留。
- 所有交互状态不得造成不必要的布局跳动。
- 颜色不能作为表达选中、错误或状态的唯一方式，应同时使用文字、图标或边框等提示。
- 关键交互应支持键盘访问，并提供清晰焦点状态；图标按钮应有可理解的名称或提示。
- 表格、表单和错误信息应具备清楚的标签与对应关系。

## 视觉基线锁定（强制）

当前已实现页面的视觉风格已经由用户确认并明确要求长期保持。除非用户明确提出“修改整体色系”“重新设计视觉风格”或对某个具体颜色作出调整，否则任何 Agent 均不得基于个人偏好、流行趋势、框架默认主题或所谓视觉优化改变现有色系和视觉语言。

本节属于硬性约束，优先级高于一般性的前端设计偏好。发生上下文压缩、摘要、模型切换或任务恢复后，必须同时重新检查本节和当前 `styles.css`，不得仅凭摘要或记忆重建样式。

### 视觉事实来源

- 根目录 `styles.css` 是当前视觉实现的唯一事实来源，`AGENT.md` 记录其不可随意改变的设计意图。
- 新增需求、新增原型页或新增组件前，必须先读取现有 `styles.css` 中的 `:root` Token 和相关组件样式。
- 应优先复用已有变量、颜色、边框、阴影、排版、间距和交互状态，不得为相同语义创建另一套近似样式。
- 若文档中的色值与 `styles.css` 不一致，以用户最近确认后的 `styles.css` 为准，并同步修正文档。
- 不得因重构、迁移框架或拆分组件而改变最终视觉结果。技术实现可以变化，视觉基线不能隐式变化。

### 设计意图

本项目是产品原型和开发说明工具，不是最终 UI 视觉稿。整体视觉必须保持克制、专业、低饱和、低品牌识别度，避免提前给 UI 设计人员造成颜色、品牌或最终视觉方案上的误导。

必须保持以下特征：

- 以白色、浅灰、灰绿色作为大面积背景和结构色。
- 以深灰绿作为主要文字、顶部栏、侧边栏和主要操作色。
- 绿色必须偏沉稳、低饱和，不使用鲜艳荧光绿或高饱和品牌绿。
- 棕金或琥珀色只用于原型组件编号角标及原型与说明的关联高亮，不扩展为大面积主题色。
- 红色只用于错误、危险或破坏性状态；蓝色主要用于键盘焦点等辅助状态，不作为页面主色。
- 状态颜色只承担必要语义，并搭配文字、边框或形状，不营造品牌氛围。
- 页面主要依靠留白、细边框、轻微层次和信息密度建立秩序，不使用强烈装饰吸引注意力。

### 固定核心色彩 Token

以下 Token 是当前视觉基线。未经用户明确批准不得替换、整体调色或改变语义：

- 主文字 `--ink: #17201b`
- 次级文字 `--muted: #66716b`
- 通用边框 `--line: #dce2de`
- 内容底色 `--paper: #ffffff`
- 原型画布底色 `--canvas: #eef1ef`
- 主操作与主要强调 `--green: #186b4d`
- 绿色浅背景 `--green-soft: #e7f1ec`
- 警示语义 `--amber: #a85f13`
- 错误或危险语义 `--red: #b6382d`
- 焦点辅助色 `--blue: #225fad`
- 轻阴影 `--shadow: 0 10px 30px rgba(28, 43, 35, 0.08)`
- 深色导航区域继续使用当前 `#18231d`、`#1c2721`、`#26362e` 附近的低饱和深灰绿，不替换成纯黑、深蓝或紫色。
- 组件编号角标统一使用 `#B70000` 背景，角标文字始终为白色 `#FFFFFF`，即使父组件处于禁用状态也不得继承灰色文字。
- 原型与说明的选中描边保持当前棕金色 `#d19a36`；说明卡片的选中强调保持主绿色 `#186b4d`。

新增语义色前必须确认已有 Token 无法表达该状态。允许为边框、悬停和浅背景使用现有颜色的低对比度邻近色，但不得借此逐步形成第二套色系。

### 禁止的视觉变化

未经用户明确要求，禁止：

- 将整体主题改为蓝色、紫色、橙色、红色、纯黑或其他强品牌色系。
- 使用高饱和色、大面积渐变、彩色光晕、玻璃拟态、霓虹效果、发光阴影或装饰性色块。
- 使用紫蓝渐变、营销落地页风格、夸张 Hero、插画式背景或与业务无关的装饰图形。
- 为每个新需求设计独立主题，导致需求之间视觉不一致。
- 使用大圆角、胶囊化容器、卡片套卡片或大面积悬浮卡片改变当前紧凑、克制的工作台风格。
- 为追求“更现代”而替换当前绿色体系、增大阴影、提高圆角或增加无业务意义的动画。
- 直接采用 UI 框架默认主题色、默认圆角、默认阴影覆盖现有风格。
- 把高保真原型做成最终 UI 视觉定稿感过强的页面；高保真指业务结构、数据、状态和交互完整，不代表强化品牌视觉。

允许保留现有需求列表页非常轻微的灰绿色背景渐变，但不得扩大渐变对比度，也不得将渐变作为主要视觉特征。

### 组件视觉一致性

- 所有需求详情沿用同一套顶部原型上下文栏、画布底色、右侧说明面板、返回按钮、说明卡片和组件角标样式。
- 主按钮使用低饱和深绿色；次按钮使用白色或浅灰底加细边框；禁用按钮使用浅灰底和灰色正文。
- 禁用状态只能影响组件本体，不得降低组件编号角标的可读性。所有深色或红色角标统一使用白色文字。
- 表单、表格和内容区域使用白底、细灰绿边框，避免明显圆角和厚重阴影。
- 选中组件时使用棕金描边及浅色背景；右侧对应说明卡片使用绿色边框或绿色内侧标记，维持当前双向识别方式。
- 错误、成功、进行中、待处理等状态应延续现有低饱和浅底色方案，不使用纯色大面积填充。
- 动画仅用于定位、加载、展开等有意义反馈，保持短促克制，不增加纯装饰动画。

### 排版与密度

- 界面默认字体继续使用当前中文无衬线字体栈；标题允许延续当前 `Georgia` 与中文字体组合，但不得引入风格强烈的品牌字体。
- 需求列表可保持较宽松的信息层级；高保真业务工作台保持紧凑、清楚、可扫描的数据密度。
- 高保真业务区字号以用户提供的实际生产后台截图为可读性基准：常规正文、表单、导航和表格不得低于 `12px`，默认优先采用 `13px`；辅助信息不得低于 `11px`；页面标题通常为 `22px` 至 `24px`。
- 右侧说明面板的规则正文和列表不得低于 `12px`，组件名称不得低于 `13px`，辅助类型和元数据不得低于 `11px`。
- `8px` 至 `10px` 仅允许用于组件 ID 角标、版本号等极短的辅助标识，不得用于需要连续阅读的业务内容、表格内容、表单标签或需求规则。
- 放大字号时必须同步检查控件高度、行高、表格行距和横向滚动，不得通过压缩字距、强制换行或缩回小字号解决布局问题。
- 不使用超大标题、负字距或随视口宽度缩放的字号。
- 说明面板文字可以紧凑，但不得以缩小字体为手段塞入过量信息；必要时使用滚动和清晰分组。

### 视觉变更协议

- 完成新增页面后，必须与当前需求列表页和 `REQ-2026-003` 订单工作台进行视觉对照，检查主色、背景、边框、角标、选中态、按钮和说明面板是否一致。

## 开发与编码规范

- 优先采用项目已有技术栈和约定；引入新依赖前应确认必要性、维护成本和部署影响。
- 使用组件化、数据驱动的结构，保持渲染逻辑与需求说明数据分离。
- 命名应清晰表达业务含义，避免晦涩缩写和炫技式实现。
- 在确有复用或复杂度收益前，不提前建立多层抽象。
- 尽可能为需求、页面、组件和说明建立集中且有类型约束的数据模型。
- 不在多个组件内重复硬编码同一业务规则。
- 代码注释只解释不明显的原因、约束和取舍，不复述代码本身。
- 保持改动范围聚焦，不顺带重构无关模块，不覆盖或回退他人已有改动。
- 不得把密钥、口令、令牌、真实个人数据或生产数据写入仓库、原型或前端包。
- 示例数据应脱敏，不得使原型误连生产接口或产生真实业务副作用。

## 数据与内容规范

- 需求、页面、组件和说明之间应通过稳定标识关联，不依赖数组位置或显示文本。
- 配置数据必须有明确的必填项和默认值，加载失败时提供可理解的错误提示。
- 删除或改名需求、页面、组件时，应同步检查导航、说明、深链接和历史引用。
- 所有用户可见文案应集中维护或易于检索，避免同一概念多种叫法。
- 示例数据应覆盖正常值、空值、边界值和异常值中与需求相关的部分。

## 测试与验收规范

每个新增或修改的需求页面至少检查：

- 需求列表入口和详情路由可访问
- 页面导航正确，刷新或直接访问链接后仍能恢复正确页面
- 每个有业务意义的组件均有 ID 和匹配说明
- 原型点击说明、说明点击原型的双向关联正确
- 默认态、加载态、空态、错误态、禁用态和成功态中适用的状态已覆盖
- 字段格式、校验、边界和日期时间精度与说明一致
- 常见桌面和移动宽度下无重叠、溢出或不可达操作
- 浏览器控制台无影响使用的错误
- 静态构建成功，线上路径和静态资源在非根路径部署时仍可用（如适用）

修复缺陷或修改共享逻辑时，应增加与风险相匹配的自动化测试。无法自动化的关键交互应记录人工验收步骤。

## 完成定义

一个需求或原型页面只有同时满足以下条件才可视为完成：

- 页面外观和关键交互已实现
- 页面级和组件级说明完整
- 所有关键组件 ID 唯一且双向关联有效
- 默认、边界、异常、权限及时间精度已明确
- `待确认` 项已解决，或经明确同意作为已知风险保留
- 相关测试和构建检查通过
- 变更记录及访问方式已更新

仅视觉上正确但缺少规则说明，不视为完成；仅有说明但关键流程无法操作，也不视为完成。

## 变更与版本管理

- 新增组件时立即分配 ID 并添加匹配说明，不在最后集中补录。
- 修改已评审的行为、字段或 ID 时，应记录变更原因、影响范围和日期。
- 需求状态、原型版本和说明内容应保持一致，避免线上展示旧规则。
- 重大调整应保留简洁变更记录，至少包括版本或日期、变更内容和负责人。
- 未经用户明确要求，不执行破坏性 Git 操作，不回退或覆盖他人的改动。

## 部署规范

- 默认目标是可重复构建和静态部署。
- 环境相关配置不得硬编码，应通过明确的配置方式管理。

- 交付时应提供线上访问地址；尚未部署时至少提供本地启动方式、构建结果和未完成原因。
